# Computational Financial Modeling Data Pipeline

Overview
This project simulates a real-world enterprise computation engine for a financial institution using SQL Server. The objective is to design, build, and automate a high-precision data pipeline that translates theoretical financial mathematics into exact computational queries. The pipeline consists of:

- A Data Layer for the ingestion of raw financial parameters and task configurations.

- A Computation Layer utilizing dynamic T-SQL logic to process algorithmic mathematical models.

- High-Precision Formatting managing strict DECIMAL data types to ensure high-fidelity financial reporting (ZAR).

- Validation Protocols to ensure query accuracy aligns perfectly with strict mathematical reference points.

- Role-based access control (RBAC) to ensure calculation integrity.

## Databases

1. **FinancialEngineeringDB**
- Central environment for all financial modeling and execution.
- Designed to handle intermediate calculations with 4-decimal precision (DECIMAL(19,4)) to prevent data loss before final reporting.

2. **Presentation Layer (Query Output)**
- The final output generated by the calculation engine.
- Formatted dynamically, enforcing a strict 2-decimal presentation for monetary values and 4-decimal presentation for rate conversions.
  
----------------------------------------------------------------------------------------------------
## Data Model
### Core Configuration Table
### dbo.FinancialTasks

- TaskID (Primary Key)
- ModelName (e.g., Compound Interest, Inflation Analysis)
- Principal (Base capital amount)
- AnnualRate (Nominal interest or depreciation rate)
- TermYears (Duration of the financial model)
- Frequency (Compounding periods per year)
  
----------------------------------------------------------------------------------------------------
## Processing Logic (T-SQL Automation)
- All financial modeling is executed using highly optimized T-SQL scripts. The execution flow prioritizes mathematical accuracy and performance:

- Schema Setup: Defines the DDL for dbo.FinancialTasks with strict data typing (DECIMAL(19,4)) to handle complex interest and decay rates.

- Data Ingestion: Standard INSERT statements provision the test cases representing different financial instruments.

- Algorithmic Execution: A master SELECT query acts as the computation engine. It uses a dynamic CASE statement to read the TaskID and apply the correct mathematical formula (e.g., POWER(1 + AnnualRate, TermYears)) on the fly.

- Precision Control: The pipeline explicitly uses CAST functions (DECIMAL(19,2)) on the final outputs to enforce GAAP-standard financial formatting without altering the underlying data.

-----------------------------------------------------------------------------------------------------
## Security & Roles
### DatabaseAdmin:
- Full control over FinancialEngineeringDB.
- Can insert new financial task scenarios, update base rates, and modify schemas.

### FinancialAnalyst:
- Read-only access to the database.
- Can execute the main computation query to generate reports.
- Cannot alter the underlying algorithms or the FinancialTasks table.

### Auditor (External):
- Strict read-only access.

## Required Analytical Outputs

- The computation engine successfully generates the following financial reports:
- Simple & Compound Interest: Calculates total interest earned and total accumulated balances over varying multi-year terms.
- Hire Purchase Analytics: Computes strict monthly installment requirements.
- Inflation Projection: Analyzes the future cost of goods over decades based on CPI trends.
- Reducing Balance Depreciation: Tracks the remaining book value of high-value company assets.
- Loan Accrual & Doubling Time: Evaluates 12-month debt interest accruals and calculates the exact timeline for capital doubling.
- Effective Annual Rate (EAR): Converts nominal monthly rates into true effective yields for cross-product comparisons.
